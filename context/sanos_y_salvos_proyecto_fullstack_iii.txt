Introducción 
El informe propone una solución para la problemática de mascotas perdidas 
mediante una arquitectura de microservicios, que permite centralizar la información, 
mejorar la gestión de reportes y facilitar la localización de animales. El sistema 
integra geolocalización, análisis de imágenes y comunicación entre usuarios para 
aumentar la probabilidad de recuperación. 
Selección de Patrones de Arquitectura y Justificación 
Además, se definen patrones de arquitectura y tecnologías que aseguran una 
solución escalable, eficiente y alineada con los requerimientos del cliente, 
considerando también aspectos de seguridad, privacidad y sostenibilidad. 
Patrones de Arquitectura Utilizados 
La arquitectura incorpora los siguientes patrones: 
● Microservicios 
● API Gateway 
● Repository Pattern 
● Factory Method 
● Circuit Breaker 
--- Page 4 ---
Estos patrones se integran dentro de una arquitectura distribuida donde cada 
servicio cumple una función específica, logrando alta cohesión y bajo 
acoplamiento entre los componentes del sistema. 
1. Patrón de Microservicios 
El patrón de microservicios divide el sistema en servicios independientes 
responsables de funcionalidades específicas. 
Microservicios principales 
● Auth Service: autenticación de usuarios 
● User Service: gestión de usuarios 
● Pet Service: gestión de mascotas 
● Geo Service: geolocalización de reportes 
● Match Service: detección de coincidencias 
● Chat Service: comunicación entre usuarios 
● Notification Service: envío de notificaciones 
● Media Service: gestión de imágenes 
Justificación 
● Permite separar responsabilidades dentro del sistema. 
● Facilita la escalabilidad de módulos críticos como coincidencias o 
geolocalización. 
● Mejora la mantenibilidad, permitiendo modificar servicios sin afectar todo el 
sistema. 
● Permite integrar servicios externos como APIs de inteligencia artificial. 
Este enfoque permite que la plataforma evolucione de forma flexible incorporando 
nuevas funcionalidades. 
2. Patrón API Gateway 
El API Gateway actúa como punto de entrada único para las solicitudes 
provenientes del frontend. 
--- Page 5 ---
Justificación 
● Centraliza las peticiones hacia los microservicios. 
● Simplifica la comunicación entre cliente y backend. 
● Permite implementar seguridad mediante autenticación y tokens JWT. 
● Reduce la complejidad del cliente evitando llamadas directas a múltiples 
servicios. 
Este patrón organiza el acceso a los recursos del sistema y mejora la gestión de las 
comunicaciones. 
3. Repository Pattern 
El Repository Pattern separa la lógica de negocio del acceso a datos. 
Justificación 
● Mantiene el código organizado y mantenible. 
● Mejora la testabilidad mediante repositorios simulados. 
Este patrón se utiliza para gestionar entidades como: 
● Mascotas 
● Reportes 
● Ubicaciones geográficas 
4. Factory Method 
El Factory Method permite crear objetos de forma flexible sin depender 
directamente de clases específicas. 
Justificación 
● Permite crear distintos tipos de reportes (mascota perdida o encontrada). 
● Facilita agregar nuevas categorías de mascotas en el futuro. 
Este patrón mejora la extensibilidad y adaptabilidad del sistema. 
--- Page 6 ---
5. Circuit Breaker 
El Circuit Breaker permite manejar fallos en la comunicación con servicios 
externos. 
Justificación 
● Evita fallos en cascada cuando un servicio externo falla. 
● Mejora la resiliencia del sistema. 
Es especialmente importante debido al uso de APIs externas de inteligencia 
artificial para análisis de imágenes.

Evaluación Parcial 2  
Implementación de Componentes Frontend y Backend 
Durante la segunda etapa del proyecto se desarrollaron e integraron nuevos 
componentes frontend y backend con el objetivo de construir una solución 
distribuida basada en microservicios, aplicando patrones arquitectónicos, estrategias 
de branching y control de versiones mediante GitHub. 
La solución fue evolucionando desde el análisis arquitectónico inicial hacia una 
implementación funcional que incorpora autenticación, geolocalización, 
administración y comunicación entre servicios. 
Componentes Backend Implementados 
La arquitectura backend fue construida bajo una estructura de microservicios 
independientes comunicados mediante APIs REST. 
1. Auth Service 
El Auth Service fue implementado como microservicio encargado de la autenticación 
y autorización de usuarios. 
Funcionalidades implementadas 
● Inicio de sesión mediante JWT. 
● Generación de Access Token y Refresh Token. 
● Validación de credenciales. 
● Protección de endpoints privados. 
● Integración con Django REST Framework y SimpleJWT. 
Tecnologías utilizadas 
--- Page 14 ---
● Django 
● Django REST Framework 
● SimpleJWT 
Justificación 
La separación del servicio de autenticación permite centralizar la seguridad del 
sistema y desacoplar el manejo de usuarios del resto de funcionalidades. 
Beneficios 
● Mayor seguridad. 
● Escalabilidad independiente. 
● Reutilización del sistema de autenticación por otros servicios. 
 
2. User Service  
El User Service fue implementado como microservicio encargado de la gestión de 
usuarios del sistema. 
Funcionalidades implementadas 
● Registro de usuarios. 
● CRUD de usuarios. 
● Panel administrativo mediante Django Admin. 
● Protección mediante JWT. 
● Serialización de datos usando DRF. 
Tecnologías utilizadas 
● Django 
● Django REST Framework 
● SQLite durante desarrollo 
Justificación 
--- Page 15 ---
Separar la gestión de usuarios facilita el mantenimiento y permite administrar la 
información de manera independiente al sistema de autenticación. 
 
Beneficios 
● Organización modular. 
● Mejor mantenibilidad. 
● Facilidad para futuras ampliaciones. 
 
3. Geo Service  
El Geo Service fue incorporado como microservicio especializado en 
geolocalización. 
Funcionalidades implementadas 
● Manejo de ubicaciones geográficas. 
● Recepción de coordenadas. 
● Integración futura con mapas interactivos. 
● Base para la visualización de reportes geográficos. 
Tecnologías utilizadas 
● Django REST Framework 
● APIs de geolocalización 
Justificación 
La geolocalización es uno de los componentes principales del sistema debido a la 
necesidad de ubicar mascotas perdidas o encontradas. 
Beneficios 
● Centralización de datos geográficos. 
--- Page 16 ---
● Escalabilidad independiente. 
● Optimización futura de consultas espaciales. 
 
 
Backend For Frontend (BFF) 
Se implementó un Backend For Frontend (BFF) como capa intermedia entre el 
frontend y los microservicios. 
Funcionalidades del BFF 
● Centralización de peticiones. 
● Simplificación de consumo desde React. 
● Comunicación con Auth Service y User Service. 
● Manejo centralizado de respuestas. 
Justificación 
El patrón BFF permite adaptar la información específicamente para el frontend, 
reduciendo complejidad y desacoplando la interfaz de los microservicios internos. 
Beneficios 
● Menor acoplamiento. 
● Mejor organización. 
● Facilidad para escalar el frontend. 
● Mayor control sobre seguridad y validaciones.

nuevos servicios como Match Service, Notification Service y Chat Service en futuras 
etapas del proyecto.
